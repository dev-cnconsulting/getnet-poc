1. Visão Geral da Integração

A Getnet (instituição de pagamento do grupo Santander) oferece a Global API — uma API REST unificada (Single Entry Point) que permite processar pagamentos online (e-commerce) e presenciais (card present) em múltiplos países da América Latina e Europa. No Brasil, a API suporta:

  • Cartão de Crédito — Visa, Mastercard, Amex, Elo e Hiper (à vista, parcelado sem juros, parcelado com juros)
  • Cartão de Débito — transações à vista
  • Pix QR Code — geração de QR Code dinâmico para pagamento instantâneo
  • Boleto Bancário — geração de boleto Santander
  • Recorrência — cobranças automatizadas com planos e subscrições
  • Split de Pagamento — divisão de valores entre múltiplos recebedores
  • Cofre (Tokenização) — armazenamento seguro de dados de cartão
  • Antifraude — scoring de risco em tempo real com IA
  • 3D Secure (3DS 2.0) — autenticação do titular com o banco emissor
📌 Documentação Oficial Toda a referência técnica está em docs.globalgetnet.com. A coleção Postman oficial está disponível na mesma documentação.

4. Autenticação (OAuth 2.0)

Todas as chamadas à API exigem um Bearer Token (access_token) obtido via OAuth 2.0 com o fluxo client_credentials. O token expira em ~1 hora (3599s).

4.1 Obter Access Token

POST /authentication/oauth2/access_token

Requisição

# Autenticação HTTP Basic: Base64(client_id:client_secret)
curl --request POST \
  --url https://api-sbx.globalgetnet.com/authentication/oauth2/access_token \
  --header 'authorization: Basic c2J4XzlmMzQzZWMzLTczNjItNGY2Yi05N2UyLTIzY2FiMjA1M2I3NDo1YTE3MDAyMC05YzMzLTQ1NTUtODM0NC1mZjNiOTY0ZDY0Mzb=' \
  --header 'content-type: application/x-www-form-urlencoded' \
  --data 'grant_type=client_credentials'

Resposta (200 OK)

{
  "access_token": "eyJ0eXAiOiJKV1QiLCJraWQiOiI1amhLMy9xK0ZpK0tTRkIrRUwwN3VhMFYwdGM9Iiwi...",
  "scope": "digital-platform-sbx:gateway-api",
  "token_type": "Bearer",
  "expires_in": 3599
}

Uso do Token

# Incluir em TODAS as requisições subsequentes
Authorization: Bearer {access_token}
x-seller-id: {seller_id}
⚠️ Gerenciamento de Token O token expira em 1 hora. A aplicação deve:
  1. Cache do access_token com TTL baseado em expires_in
  2. Renovar proativamente antes da expiração (ex: a cada 50 minutos)
  3. Em caso de erro 401 Unauthorized, renovar e retentar

6. Tokenização de Cartão (Cofre)

Permite armazenar os dados do cartão de forma segura, eliminando a necessidade de processar dados sensíveis em cada transação recorrente. Reduz o escopo de compliance PCI DSS.

POST /dpm/cofre-gw-proxy/v1/tokens/card

Requisição

{
  "card_number": "5155901222260000",
  "customer_id": "customer_21081826"
}
// → Retorna: { "number_token": "d1c81e4e-xxxx-xxxx-xxxx" }

Uso do Token no Pagamento

{
  "data": {
    "payment": {
      "card": {
        "number_token": "d1c81e4e-xxxx-xxxx-xxxx",  // Substitui "number"
        "expiration_month": "12",
        "expiration_year": "30",
        "cardholder_name": "JOAO SILVA",
        "security_code": "123"  // CVV ainda é necessário
      }
    }
  }
}
💡 Quando usar tokenização? Pagamentos recorrentes (assinaturas), "comprar com 1 clique" e checkout expresso. O cofre da Getnet armazena o PAN (Primary Account Number) em ambiente PCI compliant.

7. Pagamento via Pix QR Code

O endpoint Pix gera um QR Code dinâmico que o cliente paga via app do banco. A transação é assíncrona — a API retorna imediatamente o QR Code e o status WAITING, e a confirmação chega via webhook.

POST /dpm/payments-gwproxy/v2/payments/qrcode/pix

Headers Específicos

HeaderDescriçãoDefault
x-qrcode-expiration-timeTempo de expiração do QR Code (segundos)180 (máx: 1800)

Body — PIX Padrão

{
  "amount": 10050,          // R$ 100,50 em centavos
  "currency": "BRL",
  "customer_id": "customer_21081826",
  "order_id": "Order-00123",
  "idempotency_key": "1eb2412c-165a-41cd-b1d9-76c575d30a21"
}

Resposta (201 Created)

{
  "payment_id": "03ec0ede-3bc9-42dd-a71b-1c3a670b2b89",
  "status": "WAITING",
  "description": "QR Code successfully generated and awaiting payment.",
  "additional_data": {
    "transaction_id": "890005df15a2-0b1e-4c6e-8ece-11a71070be06",
    "qr_code": "00020101021226740014br.bcb.pix...",
    "creation_date_qrcode": "2025-10-20T18:55:55",
    "expiration_date_qrcode": "2025-10-20T18:58:58",
    "psp_code": "033"
  },
  "is_split": false,
  "idempotency_key": "e757d73e-fbdb-40ee-a72b-6f06f19c7a27"
}

O que fazer com a resposta

  1. Extrair qr_code (string "copia e cola") e exibir para o cliente
  2. Opcionalmente converter a string em imagem QR Code (biblioteca qrcode em qualquer linguagem)
  3. Armazenar payment_id para correlacionar com a notificação do webhook
  4. O QR Code expira em 180 segundos (configurável até 1800s)
🚫 Irreversível Transações Pix são processadas em tempo real e não podem ser revertidas uma vez completadas. Não há "cancelamento" — apenas estorno manual via suporte se necessário.

8. Cancelamento & Estorno

POST /dpm/payments-gwproxy/v2/payments/cancel

Requisição

{
  "idempotency_key": "new-uuid-for-cancel",
  "payment_id": "053de7f9-3725-437b-bdfc-bbf3ed0acb75",
  "amount": 118708  // Pode ser parcial
}
⚠️ Cancelamento vs Estorno
  • Cancelamento (mesmo dia): Cancela a transação antes do fechamento — devolve o limite ao cliente imediatamente
  • Estorno (dia seguinte ou mais): Devolução do valor — pode levar 2-3 dias úteis na fatura do cliente
  • Pix: Não há cancelamento automático. Estorno precisa de tratamento manual

11. 3D Secure (Autenticação do Banco Emissor)

O 3DS 2.0 (3D Secure) é um protocolo de autenticação que verifica a identidade do titular do cartão com o banco emissor. Em muitos países é obrigatório por regulação (SCA — Strong Customer Authentication).

Fluxo 3DS Simplificado

1
Init Authentication
POST /security-gwproxy/v2/enrolments-initial
→
2
Banking Login
POST /security-gwproxy/v2/enrolments-continue
→
3
Validate
POST /security-gwproxy/v2/validations
→
4
Payment
com dados 3DS no request

Endpoints 3DS

POST /dpm/security-gwproxy/v2/enrolments-initial
POST /dpm/security-gwproxy/v2/enrolments-continue
POST /dpm/security-gwproxy/v2/validations
📋 Quando implementar 3DS? Recomendado para todas as transações de e-commerce. Em produção, transações sem 3DS podem ser bloqueadas pelo antifraude. O 3DS reduz chargebacks e descontos por fraude.

12. Segurança & PCI DSS

12.1 Boas Práticas

  • Nunca armazene o PAN (número do cartão) em texto plano — use sempre a tokenização (Cofre)
  • O CVV (security_code) nunca deve ser armazenado, mesmo tokenizado
  • Toda comunicação é via HTTPS/TLS 1.2+
  • O client_secret deve ser guardado como segredo (variável de ambiente, secrets manager)
  • Implementar idempotency_key em todas as transações para evitar dupla cobrança
  • Validar assinatura dos webhooks (se aplicável) para evitar spoofing

12.2 Dados de Antifraude (Obrigatórios em Produção)

Transações sem os dados de antifraude serão automaticamente bloqueadas em produção:

14. Aspecto Comercial & Contratação

14.1 O que é a Getnet?

A Getnet é a instituição de pagamento do Grupo Santander. Funciona como uma credenciadora/adquirente — processa transações de cartão (crédito, débito), Pix, boleto e outros meios de pagamento, repassando ao estabelecimento os valores descontados das taxas acordadas. É uma das maiores credenciadoras do Brasil, com Data Center próprio certificado TIER IV (único na América Latina).

🏦 Pertencimento Getnet é uma empresa do Santander Brasil. Para integração de API (e-commerce), é uma credenciadora independente do banco — você pode ser cliente Getnet tendo conta em qualquer banco.

💵 Antecipação de Recebíveis

É possível antecipar o recebimento das vendas (receber antes do prazo padrão D+2):

  • Prazo normal: D+2 (cartão) — recebimento em 2 dias úteis
  • Antecipação: Receber em até 2 dias úteis a partir da data da venda (recebe antes)
  • Modalidade: Parcial (antecipa algumas vendas) ou Total (antecipa todas)
  • Como solicitar: App Santander Empresas, Internet Banking ou Central de Atendimento
  • Custo: Há uma taxa de antecipação (gravame/CESSÃO de crédito) — valor acordado no contrato
  • Mecanismo: A antecipação é feita por cessão de crédito dos recebíveis ao Santander, sujeita à aprovação

15. Próximos Passos

Para avançar com a POC, recomendamos a seguinte sequência:

  1. Contato comercial — Iniciar processo de acreditação com a Getnet para obter credenciais de Sandbox
  2. Configuração de ambiente — Obter client_id, client_secret, seller_id e x-transaction-channel-entry
  3. Implementação do serviço de autenticação — OAuth 2.0 com cache de token e renovação automática
  4. Implementação do pagamento por cartão — Single-step com dados de antifraude + tokenização opcional
  5. Implementação do Pix QR Code — Geração do QR Code + exibição no frontend + webhook de confirmação
  6. Implementação de webhooks — Endpoints de callback para receber notificações de status
  7. Testes em Sandbox — Validar todos os fluxos com cartões de teste e QR Code Pix
  8. Homologação — Processo formal com o time de integração da Getnet
  9. Go-live — Migração para credenciais de produção e ativação em produção
✅ Resumo do que precisamos Para processar pagamentos de Cartão de Crédito e Pix QR Code:
  • Conta comercial ativa na Getnet (contrato assinado)
  • Credenciais de API (client_id, client_secret, seller_id)
  • Servidor backend para chamar a API Getnet (as credenciais não podem ficar no frontend)
  • Endpoint HTTP para receber webhooks/callbacks de notificação
  • Integração com script de fingerprint de antifraude (fornecido pela Getnet)
  • Compliance PCI DSS mínimo (SAQ-A se usando tokenização/Cofre)
  • Cartões de teste fornecidos pela Getnet para validação em Sandbox

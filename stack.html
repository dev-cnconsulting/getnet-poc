Arquitetura recomendada para processar pagamentos: Serverless vs Provisionado

Visão Geral da Arquitetura

O sistema de pagamentos precisa de um backend que comunique com as APIs dos integradores (Getnet, Mercado Pago, Pagar.me), processe webhooks, garanta idempotência e armazene transações com segurança. A seguir, comparamos três abordagens de infraestrutura.

Opção A: Serverless (Vercel Functions / Cloudflare Workers)

⚡

Vercel Functions

Integração nativa com Next.js · Deploy automático · Cold start ~50ms · Boa para CRUD + API calls

🌐

Cloudflare Workers

Edge computing global · Cold start ~0ms · V8 isolates · Ideal para webhooks e proxies

λ

AWS Lambda

Mais flexível · Maior cold start · Integração com SQS, SNS, DynamoDB · Para escala massiva

Quando usar Serverless?

CritérioServerless
VolumeLow to medium (até ~10k transações/mês)
LatênciaImportante — edge computing reduz latency para webhooks
EquipePequena — sem DevOps dedicado
CustoPay-per-use — ideal para início (quase zero com baixo volume)
ManutençãoMínima — auto-scaling, auto-healing, sem patching de OS
LimitaçõesCold start, timeout (10-30s), sem estado persistente entre requests, conexões DB efêmeras

✅ Recomendação Serverless Para início de operações: Next.js Route Handlers na Vercel (API) + Supabase (PostgreSQL) + Cloudflare Worker para webhooks. Deploy automático, custo quase zero, escala automática. Migração fácil para Docker se volume crescer.

Nossa Recomendação para Início de Operações

✅ Cenário Recomendado — Híbrido Serverless
  1. Frontend + API: Next.js 16 na Vercel (Route Handlers para API)
  2. Database: Supabase (PostgreSQL gerenciado, pooler, auth, realtime)
  3. Webhooks: Cloudflare Worker (edge global, 0ms cold start, recebe e valida callbacks de todos os 3 integradores)
  4. ORM: Drizzle ORM (type-safe, lightweight, migrações)
  5. Cartões: Tokenização no frontend via SDK do integrador (nunca toca PAN no backend)
  6. Secrets: Vercel Environment Variables + Supabase Vault
Por que? Custo quase zero no início, deploy automático, escala automática. Quando volume crescer (>10k/mês), migra API para Docker mantendo Supabase — a arquitetura de Gateway Abstraction permite trocar de serverless para provisionado sem reescrever lógica de negócio.

Gateway Abstraction Layer — Padrão de Design

Para suportar múltiplos integradores sem acoplamento, usamos um Adapter Pattern:

// lib/payments/types.ts
interface PaymentProvider {
  name: string;
  createCharge(params: ChargeParams): Promise<ChargeResult>;
  getPixQRCode(params: PixParams): Promise<PixResult>;
  getTransaction(id: string): Promise<TransactionStatus>;
  refund(id: string, amount?: number): Promise<RefundResult>;
  parseWebhook(body: any, headers: any): WebhookEvent;
}

// lib/payments/mercadopago.ts
class MercadoPagoProvider implements PaymentProvider {
  name = 'mercado-pago';
  async createCharge(params) { /* MP Orders API */ }
  async getPixQRCode(params) { /* MP Pix API */ }
  async parseWebhook(body, headers) { /* MP webhook format */ }
}

// lib/payments/getnet.ts
class GetnetProvider implements PaymentProvider {
  name = 'getnet';
  async createCharge(params) { /* Getnet Global API */ }
  async getPixQRCode(params) { /* Getnet Pix endpoint */ }
  async parseWebhook(body, headers) { /* Getnet callback format */ }
}

// lib/payments/factory.ts
function getProvider(name: string): PaymentProvider {
  switch(name) {
    case 'mercado-pago': return new MercadoPagoProvider();
    case 'getnet': return new GetnetProvider();
    case 'pagarme': return new PagarmeProvider();
  }
}

// app/api/charge/route.ts (Vercel Route Handler)
export async function POST(req: Request) {
  const { provider, ...params } = await req.json();
  const gateway = getProvider(provider);
  const result = await gateway.createCharge(params);
  // salvar no DB, retornar resultado
}
💡 Vantagem do Adapter Pattern Permite trocar de integrador sem reescrever a lógica de negócio. Adicionar um novo gateway = criar uma nova classe que implementa PaymentProvider. O resto do sistema (DB, frontend, webhooks) não muda.

Segurança & PCI DSS

PráticaDescrição
Tokenização de cartãoUsar SDK do integrador no frontend — PAN nunca chega ao nosso backend. Reduz escopo PCI para SAQ-A
Secrets managementVercel env vars + Supabase Vault. Nunca commitar chaves no Git. Rotacionar periodicamente
Idempotency keysUUID por transação — previne dupla cobrança em retry
Webhook signatureValidar assinatura/header do webhook (cada integrador tem método próprio) antes de processar
HTTPS everywhereTLS 1.2+ obrigatório. Vercel e Cloudflare provêm SSL automaticamente
Rate limitingProteger endpoints de API contra abuso (Vercel Edge Middleware ou Upstash Ratelimit)
Audit logRegistrar todas as chamadas de API e webhooks para troubleshooting e compliance
CVV nunca armazenadoO security_code é enviado ao gateway e descartado — nunca persistido
